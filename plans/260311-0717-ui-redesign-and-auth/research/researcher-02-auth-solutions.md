# Next.js 15 Dual Authentication Research Report
**Date:** 2026-03-11 | **Version:** 1.0

---

## Executive Summary

Three viable paths exist for Next.js 15 App Router dual auth (session + API keys):
1. **Auth.js v5 (NextAuth)** - Production-ready, broad ecosystem, mature
2. **Better Auth** - Modern alternative, plugin architecture, trending adoption
3. **Lucia Auth** - Lightweight, self-hosted control, excellent SQLite/Drizzle support

**Recommendation:** **Auth.js v5** for existing projects; **Better Auth** for new greenfield.

---

## Solution Comparison Matrix

| Criteria | Auth.js v5 | Better Auth | Lucia Auth |
|----------|-----------|------------|-----------|
| **Setup Complexity** | Medium | Low | Low |
| **SQLite Support** | Via adapters | Native | Native |
| **Drizzle Integration** | Yes (adapter) | Yes | Yes (native) |
| **API Key Auth** | Credentials provider | Plugin system | Custom middleware |
| **Session Management** | JWT/Database | JWT/Database | Database-backed |
| **App Router Support** | Full (v5.0+) | Full (native) | Full (v3+) |
| **Self-Hosted** | Yes | Yes | Yes |
| **Community Maturity** | Established | Growing (post-merger) | Active |
| **Plugin Ecosystem** | Limited | Rich (2FA, passkeys, etc.) | Minimal |
| **Maintenance Status** | Stable | Recommended (2025+) | Actively maintained |

## Detailed Analysis

### 1. Auth.js v5 (NextAuth)

**Current State:**
- Stable beta (v5.0.0-beta.25+) compatible with Next.js 15 App Router
- Production-ready for session-based auth + REST APIs
- Environment variables now prefixed with `AUTH_` (not `NEXTAUTH_`)

**Dual Auth Implementation:**
```
Session Layer: JWT or database-backed sessions via middleware
API Layer: Credentials provider + Bearer token validation
```

**SQLite Setup:**
- Requires `@auth/drizzle-adapter`
- Drizzle ORM for SQLite schema management
- Standard user/session tables auto-generated by adapter

**Pros:**
- Mature ecosystem, abundant tutorials
- Works with existing systems via Credentials provider
- Flexible session strategies (JWT or DB)
- Built-in CSRF protection

**Cons:**
- Setup requires understanding of adapters/providers
- More boilerplate than newer libraries
- API key auth not out-of-box (custom Credentials provider needed)

**Suitable For:** Teams with NextAuth experience, existing infrastructure.

### 2. Better Auth

**Current State:**
- Modern successor (2025+), Auth.js team joined this project
- Recommended for new projects
- Full App Router support (native, not adapter-based)

**Dual Auth Implementation:**
- Plugin ecosystem handles multi-auth patterns
- Built-in support for API keys via custom methods
- Sessions validated via middleware (Node.js runtime in v15.2.0+)

**SQLite Setup:**
- Native SQLite dialect: `database: { dialect: "sqlite", url: "./database.db" }`
- Auto-schema generation (no Drizzle boilerplate)
- Clean, type-safe query building via Kysely

**Pros:**
- Simpler API, less boilerplate
- Plugin architecture (2FA, passkeys, magic links)
- Native Next.js App Router integration
- Auto-schema generation
- Actively funded/maintained (post-merger)

**Cons:**
- Newer ecosystem (fewer third-party integrations)
- Documentation still maturing
- Small breaking changes possible as library evolves

**Suitable For:** Greenfield projects, teams wanting modern DX.

### 3. Lucia Auth

**Current State:**
- Actively maintained (v3 stable)
- Lightweight, zero-dependency session management
- Excellent Drizzle documentation

**Dual Auth Implementation:**
- Session via database + middleware validation
- API key auth via custom route handlers
- Full control over auth flow

**SQLite Setup:**
- Native Drizzle adapter: `@lucia-auth/adapter-drizzle`
- Simple schema: user table + session table with expiresAt
- Session extension logic built-in (30-day rolling expiry)

**Pros:**
- Minimal dependencies, small bundle
- Maximum control over auth logic
- Excellent SQLite/Drizzle docs
- Session expiry management built-in
- Self-hosted with no external services

**Cons:**
- More manual implementation required
- No plugin ecosystem (build everything yourself)
- Less community tooling
- Requires understanding session mechanics

**Suitable For:** Teams wanting lightweight, full control, SQLite-first projects.

## Dual Auth Pattern (Session + API Keys)

### Recommended Approach

**Middleware Route:**
```
1. Check for session cookie → validate session
2. If no session, check Authorization: Bearer header → validate API key
3. Set req.user from either source
4. Route handler proceeds with authenticated context
```

**Database Schema Additions (SQLite):**
```sql
-- Users table
CREATE TABLE user (
  id TEXT PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  password_hash TEXT NOT NULL,
  created_at INTEGER NOT NULL
);

-- Sessions (session-based auth)
CREATE TABLE session (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES user(id),
  expires_at INTEGER NOT NULL
);

-- API Keys (agent/programmatic auth)
CREATE TABLE api_key (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES user(id),
  key_hash TEXT UNIQUE NOT NULL,
  name TEXT NOT NULL,
  last_used_at INTEGER,
  expires_at INTEGER,
  created_at INTEGER NOT NULL
);
```

**Implementation Detail:**
- Session auth for browser-based UI (Next.js server actions/routes)
- API key auth for external agents (Bearer token in REST API)
- Shared user table bridges both auth methods
- Middleware validates both, routes are auth-agnostic

## Recommendation

### For This Project (vibe-kanban)

**Choose: Better Auth** (if greenfield/starting fresh)

**Rationale:**
1. Native App Router support (no adapter indirection)
2. SQLite out-of-box (no extra dependencies)
3. Plugin ecosystem for future features (2FA, etc.)
4. Simpler API key pattern via custom methods
5. Auth.js team backing ensures long-term support

**Alternative: Auth.js v5** (if integrating with existing codebase)

**Rationale:**
1. More documentation/examples available
2. Credentials provider well-tested for custom auth
3. Broader adapter ecosystem if pivoting databases later

## Implementation Roadmap

### Phase 1: Core Session Auth
- Set up chosen library with SQLite + Drizzle
- Login/logout flow for web UI
- Session middleware protection

### Phase 2: API Key Management
- Create API key table and generation logic
- Bearer token validation in middleware
- API key CRUD endpoints (create, revoke, rotate)

### Phase 3: Dual Auth Integration
- Unified middleware handling both session + Bearer
- Route handler context aware of auth source
- Rate limiting/audit logging per API key

### Phase 4: Agent Integration
- Document API key issuance for AI agents
- Test with external agent hitting REST endpoints
- Monitor API key usage

## Unresolved Questions

1. **Password hashing strategy?** (bcrypt, argon2, etc.) - Need to confirm library defaults
2. **2FA requirement for users?** - Affects schema (Better Auth has plugins, others need custom code)
3. **API key expiry policy?** - Auto-rotate, warn before expiry, or manual?
4. **Multi-tenant support needed?** - Not covered in this research; schema may need tenant_id
5. **OAuth providers** (GitHub/Google login) - All libraries support but not needed for initial dual-auth pattern?
